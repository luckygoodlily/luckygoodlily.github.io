Docker

Docker基本介紹

主要的幾個特色

  1. 首先，Docker 是一個開源專案，提供開發、轉移和執行應用程式。
  2. 隔離性好：一台機器上可能需要跑多個服務，而服務各自需要的依賴配置不盡相同，假如說兩個應用需要使用同一個依賴，或者兩個應用需要的依賴互相有一些衝突，這個時候就需要隔離性好的環境才能解決此問題。
  3. 部屬方便：我們平常搭建環境往往會耗費我們好幾個小時的時間，因為每台主機環境不盡相同，遇到的問題又會不一樣。
    而有了container之後，建置環境非常容易且快速，你的開發環境就只是一個或者幾個container，最多再需要一個控制部署流程的執行腳本像docker-compose就能容易架設起環境。
  4. 快速回滾機制：類似git的機制，容器技術天生帶有回滾屬性，因為每個歷史容器或者鏡像都會保存，而替換一個容器或者某個歷史鏡像是非常快速和簡單的。
  5. 建立和維運管理成本低：這也是容器技術發展迅速的最主要原因。伺服器的購置成本和運維成本都很高，而vm需要占用很多不必要的資源。相比之下，容器技術省去搭建作業系統的成本，

Docker 想解決的問題：

改善傳統虛擬機器因為需要額外安裝作業系統，導致啟動慢、佔較大記憶體的問題

Docker 提供的解法：

以應用程式為核心虛擬化，取代傳統需要Guest OS 的虛擬化技術

Virtual Machine VS Container

  • VM虛擬化是實作在硬體層之上，建立一個可以執行完整作業系統的獨立環境。

  • Container是實作在作業系統層之上，應用程式本身與相關的函式庫、環境配置檔，打包成映像檔 (Image) ，以達到應用程式的獨立與模組化。

    簡而言之，VM打包作業系統，Container打包應用程式。

Docker 底層實作技術

Docker 的底層是使用 Linux Kernel的 Namespace 和 Cgroup 組合而成的。
Namespace： 提供了 Container 彼此之間與作業系統是完全隔離且不受影響的，像是把Process ID、User ID、Network…等等的環境執行狀態隔離開來。如左圖例子來說在一台實體機器上，有 2個 Namespace。Namespace 1 裡面的 PID 有 1，Namespace 2的PID 也可以有 1 的 PID，透過這樣的機制做到執行環境上Process ID的隔離。
Cgroup：用來做系統資源的管理，如：CPU、Memory、I/O…等等，資源做使用上的限制和優先順序。
有了 Namespace 和 Cgroup 後，就可以做到在同一個 Kernel的系統環境下，有效的隔離資源的使用。

Docker 基本架構

主要元件：

平常大家在講的「Docker」，其實是指 Docker Engine，他是使用主從式架構(client-server 模式)
如上圖我們可以看到docker的架構

  • 在最裡層Docker daemon:它是一個長期在背景運行(long-running)的 server，用來執行管理 Docker image，提供 Restful API 給使用者操作，和顯示 Docker container 的狀態訊息
  • REST API：就是遠端伺服器，使用者需要時可將image載入於本機中。
  • docker CLI：主要是提供指令操作，透過REST API的方式與Docker daemon溝通。

docker CLI 使用範例：

  1. docker build ：如左圖我標的路徑1，他會先去命令 daemon 建立一個應用程式的 image。
  2. docker pull ：如路徑2，命令 daemon 從 registry 下載 image。
    registry 是存放 image 的倉庫，若要分享 image 給別人，可以 docker push 把 image 上傳到 registry。 docker 預設公開的 registry 會放在 Docker Hub上 。也可以架設一個私人的 registry。
  3. docker run ：如路徑3，命令 daemon 以 image 為範本建立並執行 container。

Docker 的基本物件：

主要有container image network和volume

  • Docker Image 是映像檔(只能讀取的檔案，不能改寫):含蓋Container內的所有程式(包括應用程式、函式庫、環境設定檔)，可以用來建立Docker Container，使用者想要什麼樣的執行環境也可以到DockerHub上找到相對應的映像檔載下來。例如：完整的 MySQL 服務或是一個 Ubuntu 作業系統等。
  • Container(可以改寫的部分): 是根據docker image建立的instance。每個container之間都是互相獨立的平台。容器是一個簡易版的 Linux 環境應用程式並可在其中運行程式。
  • Network :讓外界或其他容器可以存取容器中的應用程式
  • Volume: 獨立於容器的檔案空間，當需要「共用」和「留存」資料時會用到 Volume。

Network

network解決的問題:
container是封閉的空間，但有時我們會需要讓其它人可以存取容器中的應用程式，甚至讓各個 container串接起來溝通時，會需要有網路的設置

Docker的網路分為以下幾種模式:

  1. Bridge Mode
  2. Host Mode
  3. None Mode
  4. Container Mode
Bridge Mode

預設的network，容器啟動時會創建一對veth pair(虛擬網路接口)設備，Docker會將一端掛載到 bridge網路物件上，另一端放入容器的Network Namespace內，從而實現容器與主機通信的目的。

如左圖，因為 veth2讓 bridge(172.17.0.1) 與 eth0(172.17.0.2) 位於同一個區網。

Container 2 可透過 eth0 經過 veth2 與 bridge 進行溝通，且 Container 1 與 Container 2 透過IP進行溝通。
適合對網路沒有要求，單節點彼此能通的使用者使用

Host Mode

Host 模式相對簡單些，可以把他想像成建立一個與Host 擁有同樣的 network interface 的 Container
圖中不一樣的是只有一個eth0 的 network interface ，所以每個容器可以用 192.168.0.2: 通訊。

通常我們的使用情境是：
容器需要使用特殊硬體資源，但掛載到容器導致相對麻煩時，我們則會使用這種方式共用宿主機的資源
但因為與主機共享Root Network Namespace，容器有完整的權限可以操縱主機的protocol、port和防火牆等，所以被認為是不安全的方式。
也要特別注意 Port Number 的使用，若啟用相同服務的port時就很容易發生衝突

None Mode


None Mode最簡潔的模式，沒有任何 network interface 的 Container，也就是不為容器配置任何網路功能，但仍然可以手動為容器配置網路。

適合想要開發網路模型，或想要研究 Docker 網路，以及想要開發 CNI (Container Network Interface)的人都適合用這種模式創建乾淨網路。

Container Mode


和其他三個種網路建立方式不一樣，前面三個只要直接run container就可以了。

而container Mode的建立方法:
以bridge模式建立一個container，並且使用這個 Container ID 建立另外一個 Container

特色:
與另一個運行中的容器共享相同的網絡，所以可以使用 localhost 來存取彼此服務。
如左圖雖然兩個 Container 的 IP 都是 172.17.0.2 ，雖然他們是不同的 Container 但是被放置同一個 Namespace 內，可以透過此方式享受到更快的網路存取速度

Volume

Docker 提供三種掛載類型(mount types)：

  • bind mount(與 host 共享)：掛入現有的 host 檔案系統(filesystem)，用在容器與 host 共享資料夾或檔案
  • volume(容器掛載 Volume 實體)：掛入 volume 物件
  • tmpfs mount：從 memory 掛載

Dockerfile和docker-compose之間差異

Dockerfile內容是把指令式的模式寫在文字檔中，主要是用來描述image的文件，可以幫助我們建立和執行image，在建置環境上減少重複的步驟

而Docker Compose用於快速建立多個container的Docker工具，也就是描述一個service應該如何組成。
Docker Compose是在docker-compose.yml檔案中定義其構成的應用服務和所有依賴服務的container，藉此讓它們可以在單一隔離環境中一起運行。
