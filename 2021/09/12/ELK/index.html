ELK介紹

什麼是ELK?

ELK 是指 Elasticsearch(E) 、Logstash (L)和Kibana (K) 這三個開源軟體的集合套件，這三個軟體可以組成一套日誌 ( log ) 分析和監控架構。

ELK 的起源：Elasticsearch（NoSQL資料庫）

  • Elasticsearch 是一個及時分散式的資料庫兼檢索引擎
  • 提供 REST API 方便大家操作
  • 可以用在任何需要把資料儲存並快速進行全文檢索
  • 搜尋引擎執行可以優化查詢的速度

架構圖

ELK 是由 Elasticsearch、Logstash 及 Kibana 三個系統所組成的 Log 蒐集、分析、查詢系統。

  • Logstash:是日誌採集的中心，負責日誌解析與處理，會把所有採集的log都傳遞到logstash中。
  • Elasticsearch:本身是一個資料庫用json的格式做存取，用了一個全文索引可以查詢非常大量的資訊。
  • Kibana:是圖形化的展示，可以用來展示ES中的數據的
  • Beats 是Log採集工具，負責將本台電腦的Log檔取出傳遞到logstash做過濾，或是直接傳遞到ES儲存器起來。

什麼是 Filebeat

Filebeat是Beats中的一員。Beats在是一個輕量級日誌採集器，目前Beats包含六種工具：

  • Packetbeat：網絡數據（收集網絡流量數據）
  • Metricbeat：指標（收集系統、進程和文件系統級別的CPU和內存使用情況等數據）
  • Filebeat：日誌文件（收集文件數據）
  • Winlogbeat：windows事件日誌（收集Windows事件日誌數據）
  • Auditbeat：審計數據（收集審計日誌）
  • Heartbeat：運行時間監控（收集系統運行時的數據）

早期的ELK架構中使用Logstash收集、解析日誌，但是Logstash對內存、cpu、io等資源消耗比較高。相比Logstash，Beats所佔系統的CPU和內存幾乎可以忽略不計。

Filebeat以agent的角色監控指定的log files，收集log events，然後forward至Elasticsearch或Logstash來建立索引

Filebeat如何運作的

Filebeat可以包含一個或多個input，而每一個input可以指定多個log data locations，

filebeat針對每一個log都會有一個收割程序(harvester)，每個收割程序會讀取log的新內容，
如有新的內容，併發送這些新的日誌資料到處理程式（spooler），處理程式會集合這些事件，然後再送至設定的output，

output可以是es或Logstash

filebeat.yml 內容

Input

支援複數 inputs，filebeat 去讀取這些 log 檔案，並且在系統上紀錄讀取的進度，偵測到 log 有增加，變繼續讀取新的 log。

  • Type: 預設類型是log類型
  • Enabled:設為true表示啟用這個block的input功能
  • Scan_frequency:表示間隔多久掃描一次
  • Tags and fields:可以加入一些自己想記錄的特性，這裡我是紀錄server hostname
output

output 也是 filebeat 十分重要的一環，好的 filebeat output 設定，可以大幅降低整體 ELK stack 的負擔。

  • logstash: 先向後送到 logstash, enable設定為true開啟功能，並指定hosts
  • elasticsearch: 直接向後送進 elasticsearch,設定elasticsearch的host並設定index的名稱
  • Kibana:最後設定kibana的host,指定elasticsearch的index要送去的位置
modules

Filebeat有很多modules，裡面已經包含了很多默認模板，你可以直接使用默認設置從系統默認路徑中抓取文件，再進行下一步，將我們的輸出減少到logstash，做管道預處理非常方便。

什麼是 Logsatsh

logstash.conf的大致架構


logstash幫助你去收集各地的log或是資訊，並且根據你的格式，轉換成各種資料欄位．

Logstash包含3個主要部分： 輸入(inputs)，過濾器(filters)和輸出(outputs)。

  • Inputs:由各種地方抓取相關的資料，不論是標準輸入(stdin)，檔案或是beat都可以．
  • Filter:抓取到資料之後，這時候可以選擇使用grok去拆解資料．比如說把一串非常長但是沒人看得懂的log拆解成各個相對應得資料欄位．
  • Output:可以輸出到各種地方(一次可以多個輸出)，不論是輸出到stdout印出來，或是直接丟給Elasticsearch拿來做分析用甚至可以直接輸出到資料庫．

Input-plugin

Logstash 輸入插件常用的是Beats系列（例如Filebeat 和Metricbeat），還包含各種file，Amazon s3。Logstash 當前支持50多個輸入插件

  • Beat:

    Logstash 匯總Beats 收集的日誌，並使用有效的Logstash 工作線程創建有效的批次處理，這可以大量減少索引操作的次數。因而減少Elasticsearch 的負擔。
    右邊的例子表示了，輸入配置告訴Logstash監聽5044端口上的Beats事件，並將它們直接發送到Elasticsearch。

  • File:

    此插件用作文件文字檔的監視程序，將文件末尾的新行視為新事件。此功能好處是，該插件將當前位置存儲在它跟踪的每個文件中，這樣它就可以從停止和重新啟動Logstash 時從中斷處開始。
    我這裡簡單介紹file的參數設定
    path 讀取的監聽文件的絕對路徑，可以使用匹配模式
    Discover_interval => 15 #設置多長時間掃描目錄，發現新文件
    start_position => “beginning”
    logstash從什麼位置開始讀取文件數據，默認是結束位置，也就是說 logstash 進程會以類似tail -F的形式運行。 如果需要從頭取得全部 log 內容則設成beginning

  • Amazon S3

    預設logstash 沒有安裝s3 input plugin，所以我們首先需要在container中安裝plugin，撰寫在docker-compose中，我們會下載後並指定要執行哪個config檔案

    access_key_id, secret_access_key, region和bucket是s3需要的參數
    prefix是要放入s3的檔案路徑，支持 dynamic parameters, Ex: "BB/%{[fields][log_source]}/%{+YYYY}/%{+MM}“
    codec 在數據進入輸入之前對其進行json 解碼，而無需在 Logstash 管道中使用單獨的過濾器。
    backup_to_dir 將處理過的文件備份到的本地目錄的路徑。

Filter

Filter是負責把收到的純文字資料，做特定的規則處理，就可以變成指定的欄位。 建立欄位的好處是可以方便搜尋，而且也能做到比全文檢索更好的分析，

  1. Grok
    Grok Patterns 的基本用法是：%{Pattern名稱:欄位名稱:型別}
    Type default: text
  2. Geoip
    GeoIP 過濾器根據 MaxMind GeoLite2 數據庫中的數據添加有關 IP 地址地理位置的信息。
  3. Mutate
    它提供了豐富的基礎類型數據處理能力。包括，字符串處理和增加或刪除欄位和類型行別轉換等。
  4. Date
    根據grok切出的時間(HTTPDATE)來改成新個格式或時區

Output-plugin

  • elasticsearch
    解析完資料新建index放入elasticsearch
    – Host: 可設定多台elasticsearch，預設值是[127.0.0.1]

    – Index:設定建index的名稱，不能有大寫字母，並支持動態語法

    – document_id:每個索引都有唯一文檔id，我們可以使用這個id來防止重複數據導入

    指紋過濾器將原始事件中的一個或多個字段（默認為message）作為原來創建一致的哈希值(hash)，將其用作Elasticsearch輸出中的文檔ID。

  • Amazon S3
    預設logstash 沒有安裝s3 input plugin，所以我們首先需要在container中安裝plugin，撰寫在docker-compose中，我們會下載後並指定要執行哪個config檔案

    access_key_id, secret_access_key, region和bucket是s3需要的參數
    prefix是要放入s3的檔案路徑，支持 dynamic parameters, Ex: "BB/%{[fields][log_source]}/%{+YYYY}/%{+MM}“
    codec和input用的不一樣，需要使用json_line
    temporary_directory:暫存傳上去s3的資料

什麼是Elasticsearch

Elasticsearch Features

  • ElasticSearch (簡稱 ES) 是一個分散式搜尋引擎
  • ES 是一個提供全文字搜尋引擎功能的開源 Java 專案，
  • 其底層採用了 lucene 全文字搜尋框架，lucene 為 ES 提供 invert index，文件儲存以及文件搜尋的功能，
  • 而 ES 在 lucene 上提供出 cluster 架構，讓使用者能透過多台電腦中的硬碟來處存大量的文件。
  • ES 還提供 HTTP 接口，讓使用者可用不同語言實作 client-side 程式來與 ES 交互，
  • 透過 DSL 設計自己的查詢語法，來做到不單只有全文字的搜尋，還可以做得資料聚合的功能
  • 在全文字搜尋上，ES 提供可擴充的 Analyzer 模組，主要用於文件中取文字以及標準化的邏輯，因此我們可以透過自己訂義的 Analyzer 來實作中文版的全文字搜尋引擎。

Document & JSON document

  • Document
    – Document 是搜尋數據的最小單位
    – Document 會被序列化成 JSON(由一堆 Key/Value 的資料組成，並有其資料格式) 格式，保存在 Elasticsearch 中
    – 每個 Document 都有一個 Unique ID，可自己指定或交由 Elasticsearch 自動產生
  • JSON document
    – 包含多個 Key/Value 組合，就像是資料庫中的一筆資料
    – 但跟資料庫不一樣的是，JSON 格式靈活不受限，不須預先定義格式
    – 每個 Key/Value 的類型(string, number, boolean … etc) 可以自己指定或是由 Elasticsearch 幫忙推算

Metadata & Index

  • document metadata :描述 document 本身屬性用的資料
  • index: 在 ES 中是個邏輯空間的概念，用來儲存 document 的容器，而這些 document 內容都是相似的
  • shard: 在 ES 中則是個物理空間的的概念，index 中的資料會分散放在不同的 shard 中

index 組成：

  • data：由 document 和 metadata 所組成
  • mapping：用來定義每個欄位名稱 & 類型
  • setting：定義資料是如何存放(例如：replication 數量, 使用的 shard 數量)
  • 在 ES 7.0 的版本後，index 在 type 部份只能設定為 _doc (在以前的版本是可以設定不同的 type)

Kibana

Kibana 是虛擬開源的數據分析和可視化平台，您可以使用 Kibana 對 Elasticsearch 索引中的數據進行搜索、查看、交互操作。
基於瀏覽器的界面便於您快速創建和分享動態數據儀表板來追踪 Elasticsearch 的實時數據變化。

kibana解決得痛點

  1. 資料視覺化:kibana可以很方便的利用圖表、表格及地圖對數據進行多元化的分析和呈現。
  2. Kibana Lens 提供簡單直觀的拖拉式視覺化工具
  3. Dashboards: 提供預設範本也可自訂和編輯
  4. Alert 機制: 設定 thresholds 來觸發特定動作 (Slack 通知、寄信等等) 這個功能在免費版中沒有
